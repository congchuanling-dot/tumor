# Tumor 功能规划（小而精 · 精准对标企业/真实开发）

原则：**不加臃肿功能，只做「开发/测试/预发里立刻用得上」的增强**。以下按优先级与实现成本分类。

---

## 一、故障类型扩展（精准补缺口）

新增故障类型时建议**延续 Tumor 隐喻**，便于叙事统一、好记好讲：

| 功能 | 肿瘤隐喻（建议） | 说明 | 企业场景 |
|------|------------------|------|----------|
| **@SlowQuery** | **肠道淤塞型**（肿瘤堵塞管道，流通变慢） | 模拟 DB/JDBC 慢查询（方法内 sleep 或占用连接一段时间） | 验证连接池、事务超时、从库切换、慢 SQL 告警是否生效 |
| **@ConnectionExhaust** | **占位型**（肿瘤占满关键通道，正常细胞过不去） | 模拟连接池被占满（占住 N 个连接不释放一段时间） | 验证连接池配置、熔断、降级到缓存/备用 |
| **入参影响故障** | **诱因敏感型**（某些刺激下肿瘤更活跃） | 如 `delay = "request.amount > 100 ? 2s : 200ms"`（SpEL 或简单规则） | 大订单/大查询更大概率慢或失败，贴近真实热点故障 |

保持「注解 + 少量参数」，不引入复杂 DSL。

---

## 二、安全与可控（企业必选项）

| 功能 | 说明 | 价值 |
|------|------|------|
| **注入频率上限** | 全局或按方法：每分钟最多注入 N 次（`tumor.rate-limit-per-minute`） | 防止误开或压测时注入爆炸，拖垮环境 |
| **注入总预算** | 单实例生命周期内最多注入 M 次，用满即停（可选配置） | 预发/演练时「只允许搞坏 100 次」心里有数 |
| **生产环境硬开关** | 当 `spring.profiles.active=prod` 时强制忽略所有注入（即便 enabled=true） | 多一层保险，避免配置错误带到生产 |

实现轻量：在切面里用原子计数器 + 配置即可。

---

## 三、流量维度的精准注入（真实需求）

| 功能 | 说明 | 企业场景 |
|------|------|----------|
| **按请求头触发** | 仅当请求带 `X-Chaos: true`（或可配置 header）时才注入 | 生产可开着 Tumor，只有带特定 header 的压测/演练流量会触发，日常流量不受影响 |
| **按采样率** | 如 1% 的请求注入延迟，其余正常 | 灰度/金丝雀场景：少量流量走故障路径，验证监控与熔断 |
| **按调用方** | 仅当调用栈或请求头里是「测试服务/内部 IP」时才注入 | 仅内部联调、压测调用触发，外部请求不受影响 |

做到「开关 + profile + header/采样」三重控制，满足企业「要可控、可审计」的要求。

---

## 四、可观测（小集成，不重）

| 功能 | 说明 | 价值 |
|------|------|------|
| **Micrometer 指标** | 暴露 `tumor.injections`（counter，tag: type=latency|starvation|...） | 和现有 Grafana/Prometheus 打通，看「今天演练触发了多少次」 |
| **可选 SLF4J 结构化日志** | 每次注入打一条带 type、method、params 的日志，便于日志平台检索 | 排查「这次请求为什么慢」时能直接搜到是否 Tumor 注入 |
| **Actuator 只读端点** | 如 `GET /actuator/tumor` 返回当前实例哪些类/方法打了 Tumor 注解（不暴露开关状态） | 运维/ SRE 知道「这台机器哪些地方可能被注入」，便于演练对照 |

不自己做监控大盘，只「接出去」，保持小而精。

---

## 五、测试与 CI 友好

| 功能 | 说明 | 价值 |
|------|------|------|
| **测试里强制启用** | 在单元/集成测试中 `@Import(TumorTestConfig.class)` 或 profile=test 时自动 enabled=true | CI 里跑「带混沌」的集成测试，验证熔断/降级路径 |
| **Resilience4j/Spring Retry 示例** | 官方文档 + tumor-demo 里一段「Tumor 注入延迟 + Resilience4j 超时/熔断」的完整示例 | 开发者照抄即用，立刻看到 Tumor 与现有容错组件的配合效果 |
| **断言工具（可选）** | 如 `TumorAssertions.assertUnderChaos(()-> service.call(), throwsException(...))` | 测试里显式断言「在混沌条件下应抛异常/应返回降级结果」 |

重点放在「文档 + 一个完整 demo 场景」，而不是做大而全的测试框架。

---

## 六、动态控制（轻量）

| 功能 | 说明 | 价值 |
|------|------|------|
| **配置刷新** | 与 Spring Cloud Config / Nacos 等兼容：改 `tumor.enabled` 或 probability 后 `@RefreshScope` 或配置中心推送生效 | 预发/演练时改配置即可，无需重启 |
| **Actuator 开关（可选）** | 如 `POST /actuator/tumor?enabled=false`，需配合 Actuator 安全（仅内网或鉴权） | 临时关闭本实例注入，便于排查问题 |

不自己做配置中心，只「接标准 Spring 配置 + 可选 actuator」，实现成本低。

---

## 七、明确不做的（守住小而精）

- **不做**独立 Tumor 控制台/大盘：用现有监控 + 指标/日志即可。  
- **不做** K8s/基础设施级混沌（杀 Pod、断网）：只做**代码/方法级**。  
- **不做**复杂编排（多服务联动故障）：单机、单方法维度足够。  
- **v2 再考虑**：ByteBuddy Agent 模式（无 Spring、纯 Java）、或与 OpenTelemetry trace 的简单关联（在 span 上打 tag 表示本次请求被注入）。

---

## 八、建议实施顺序（按价值/成本比）

1. **v1.1**：安全硬开关（prod 强制禁用）+ 注入频率上限 + 按请求头触发（如 `X-Chaos: true`）。  
2. **v1.2**：@SlowQuery + Micrometer 指标 + 文档里 Resilience4j 示例。  
3. **v1.3**：采样率 + 入参影响故障（简单 SpEL 或配置）。  
4. **v1.4**：Actuator 只读端点 + 可选配置刷新/actuator 开关。  
5. **v2**：ByteBuddy Agent、非 Spring 场景（按社区反馈再定）。

以上均围绕「代码级、注解即用、测试/预发验证容错」的定位，不扩成平台级产品，保持小而精、精准瞄准企业真实开发与演练需求。
